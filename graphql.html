Overall Idea

  • The server defines a Schema with all possible fields a client can query and mutate
  • The client defines Queries and Mutations to interact with the backend against the Schema

Types & the Schema

Scalar Types

  • String

    A UTF‐8 character sequence.

  • Int

    A signed 32‐bit integer.

  • Float

    A signed double-precision floating-point value.

  • Boolean

    true or false.

  • ID

    The ID scalar type represents a unique identifier, often used to refetch an object or as the key for a cache. The ID type is serialized in the same way as a String; however, defining it as an ID signifies that it is not intended to be human‐readable.
Source 
                                    
type User {
   name: String
   age: Int
   weight: Float
   isAdmin: Boolean
   id: ID
}

Object Types

Object Types can reference other Object Types

Fields marked with a ! are mandatory/required (non-nullable)

users: [User!]! - an Array of users which contains no nulls, neither the array itself can be null nor the contents of the array

In contrast users: [User]! - an Array of users which contains 0 or more Users as well as null

users: [User] - the whole field might be null

                                    
type User {
  name: String
  email: String
  age: Int
}

type Tweet {
  text: String!
  author: User!
}

type TweetList {
  users: [User!]!
}

Enums :yay!:

Enumeration types are a special kind of scalar that is restricted to a particular set of allowed values.

This can give you some guarantees about the returned values.

                                    
enum UserType {
  ADMIN
  MODERATOR
  MEMBER
}

Interfaces

An Interface is an abstract type that includes a certain set of fields that a type must include to implement the interface.

                                    
interface Human {
  age: Int!
  height: Int!
}

type SuperHuman implements Human {
  age: Int!
  height: Int!
  superPowers: [String!]!
}

type Villain implements Human {
  age: Int!
  height: Int!
  childHoodTraumas: [String!]!
}

Queries

Queries

  • The result of a query will have the same shape as the query itself
  • Queries can have comments, starting with "#"
  • You can pass parameters to queries
  • Each field and nested object can receive parameters

Fragments

Fragments can help you to dry out your queries

                                    
query {
  user(id: 12) {
    ...basicInfo
  }
}

fragment basicInfo on User {
   email
   name
   age
}
                                    
                                
                                    
{
  "user": {
    "email": "foo@example.org",
    "name": "JDoe",
    "age": "42"
  }
}

Variables

Never do string interpolation from user supplied values, use Query Variables

                                    
query User($userID: Int) {
  user(id: $userID) {
    id
    email
    name
    age
  }
}
                                    
                                
                                    
# query variables
{
  "userID": 1
}
                                    
                                
                                    
{
  "user": {
    "id": 1,
    "email": "foo@example.org",
    "name": "JDoe",
    "age": "42"
  }
}

Directives

                                    
query User($userID: Int) {
  user(id: $userID) {
    id
    email
    name
    followers @include(if: $withFollowers) {
      name
    }
    tweets @skip(if: $shortSummary) {
      text
    }
  }
}
                                    
                                
                                    
{
  "withFollowers": true,
  "shortSummary": false,
  "userId": 1
}

Directives enable you to conditionally include or exclude payload. The spec defines @include(if: ...) and @skip(if: ...).
Servers can implement and define custom directives.

Mutations

Mutation Types

At some point in time, you want to change state on the server. Mutation Types enable you to do so. While query fields are executed in parallel, mutation fields run in series, one after the other. 

                                        
mutation ($author: User!, $text: String!) {
  createTweet(author: $author, text: $text) {
    author {
      name
    }
    text
  }
}
                                        
                                    
                                        
# input
{
  "author": {
     id: 12
  },
  "text": "Greeeeaaaatttt"
}

Input Types

Like Type Objects, but specifically to be passed as input variables, e.g. for mutations and creating whole new objects. 

                                    
input TweetInput {
  author: User!
  text: String!
}

mutation ($tweet: TweetInput!) {
  createTweet(tweet: $tweet) {
    author {
      name
    }
    text
  }
}
                                    
                        
                                        
# input
{
  "tweet": {
     "author": {
        id: 12
     },
     "text": "Greeeeaaaatttt"
  }
}

Conclusion

Advantages - when to use

  • Get exaclty what you ask for

    Highly specific queries, only receive what you requested.

  • If bandwidth is expensive (mobile apps)

    Save data transfer and avoid multiple round trips to the backend, due to point 1.

  • Proxy several (REST) APIs behind a single interface

    Integrate several services into one API.

  • If you are serving more than one end device (web, iOS, Android)

  • If you are working on a greenfield project

    Because why not? =)

When you might want to avoid it

  • If you have full control over your backend

    If it is quite simple and/or a monolith.

  • If you are serving a public API

    REST is still the de facto standard here.

  • If you can live with blown (due to multi purpose) API responses

    Think of internal APIs that are only available within your datacenter or office.
